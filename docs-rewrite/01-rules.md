# Rewrite Rules

## Audience and Tone

1. Write for real users (attendees, moderators/chairpersons, admins), not developers.
2. Be explicit and procedural; avoid vague guidance.
3. Use plain language and short sections.

## Required Structure for User Guides

Each task-oriented page should usually include:

1. `Who this is for`
2. `Before you start`
3. `Step-by-step`
4. `What you should see`
5. `If something goes wrong`
6. `Technical note` (only when needed)

## Step Quality Rules

1. Use numbered steps for actions.
2. One user action per step where possible.
3. Mention exact UI control labels users should click.
4. Mention expected state changes after critical steps.

## Content Rules

1. Prefer user outcomes over system internals.
2. Do not expose routes/endpoints unless needed for support/operations sections.
3. Keep implementation details in short callouts, not the main flow.
4. Add cross-links to related tasks instead of duplicating long explanations.

## Media Rules (Screenshots/GIFs)

1. Use captures that match the described flow exactly.
2. Prefer highlighted focus areas for action-heavy screens.
3. Use GIFs only for transitions/interactions where motion matters.
4. Avoid decorative media that does not explain a step.

## Localization Workflow

1. Rewrite English first.
2. Mirror structural changes in German for the same file before marking complete.
3. Keep term mapping consistent (track terminology changes in decisions log).

## Review Checklist

1. Could a first-time user complete the task using only this page?
2. Are prerequisites explicit?
3. Are edge cases/failures covered for key steps?
4. Are screenshots/GIFs current and correctly highlighted?
